# 公認画家プロフィール入力・承認システム 実装指示書

公認画家が契約時に専用URLからプロフィール情報を入力し、管理者がレビュー・承認したうえでARTISTページに反映する仕組みをWordPressプラグインとして実装する。

**重要：公開事故を防ぐため、承認後も「下書き保存」までを自動とし、最終的な「公開」ボタンは管理者の手動操作とする二段階フローを採用。** 特に本名・住所・振込先などの個人情報が誤って公開される事故を絶対に防ぐ設計とする。

画像（プロフ写真・作品画像）は管理者側で別途追加する運用とし、フォームではテキスト情報のみ扱う。

---

## 1. 全体仕様

### ユーザーストーリー

1. **管理者**：契約成立時に管理画面から「画家招待」を実行 → 専用トークンURLが発行され、画家にメール送信される
2. **画家**：受信したURLを開いて、フォームに必要情報を入力 → 送信
3. **管理者**：管理画面に「未承認の入力」通知 → 差分表示で項目ごとに承認/差戻しを判断
4. **承認された項目**が `artist` カスタム投稿の **下書き** として保存される（**この時点では公開されない**）
5. **管理者**：WordPress標準の投稿編集画面で改めて下書きを確認 → プロフ写真・作品画像を追加 → 「公開」ボタンを手動クリック
6. ARTISTページに反映・サイト上に公開

### 二段階フローの設計意図

「承認 = 即公開」にすると、確認漏れで本名・住所など非公開項目が誤って公開される事故が起きる。これを防ぐため：

- **承認操作**：画家入力データをartistカスタム投稿の「下書き」に転記するだけ
- **公開操作**：WordPress標準の投稿編集画面で「公開」を別途実行（管理者が改めて全体を確認したうえで）
- これにより、承認と公開の間に必ず人間のチェックが入る

### 機能要件

- **トークン付URL発行** — 管理者が招待ボタンで一意トークンを生成、メール送付
- **画家入力フォーム** — 公開URL、ログイン不要、トークン検証のみ
- **初回入力のみ** — 一度送信したら画家側からは編集不可（修正は管理者に連絡）
- **差分承認** — 項目ごとに承認/差戻し（一括承認も可）
- **プレビュー機能** — 管理画面で「実際のARTIST個別ページ風」に表示して確認
- **承認後の反映先は「下書き」固定** — `post_status='draft'` で保存、公開は手動

### 開発スタック

- **環境**：ConoHa WING + WordPress
- **言語**：PHP（WordPressプラグイン形式）
- **依存**：ACF Pro（既存の `artist` CPTで使用想定）、jQuery（管理画面UI用）
- **メール**：wp_mail()
- **開発者**：Claude Code

### 関連する既存仕様

- `artist` カスタム投稿タイプ（既存）
- ACFフィールド：氏名、活動名、略歴、制作テーマ、なぜ描くか、SNSリンク等（既存）
- 画像（プロフ写真・作品画像）は本フォームでは扱わない（管理者がメディアライブラリから別途追加）

## 2. データベース設計

### `wp_boa_artist_submissions` テーブル

| カラム | 型 | 説明 |
|---|---|---|
| `id` | BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY | 提出ID |
| `token` | VARCHAR(64) NOT NULL UNIQUE | URL用トークン（256bitハッシュ） |
| `invited_email` | VARCHAR(255) NOT NULL | 招待先メールアドレス |
| `invited_name` | VARCHAR(100) NOT NULL | 招待先の表示名（管理用メモ） |
| `artist_post_id` | BIGINT UNSIGNED DEFAULT NULL | 紐づくartist投稿ID（承認後に確定） |
| `status` | ENUM('invited','submitted','partial_approved','draft_saved','rejected') NOT NULL DEFAULT 'invited' | 状態 |
| `submitted_data` | LONGTEXT DEFAULT NULL | 画家入力データ（JSON） |
| `submitted_at` | DATETIME DEFAULT NULL | 入力提出日時 |
| `reviewed_by` | BIGINT UNSIGNED DEFAULT NULL | レビューしたWPユーザーID |
| `reviewed_at` | DATETIME DEFAULT NULL | レビュー完了日時 |
| `review_notes` | TEXT DEFAULT NULL | レビュー時の備考 |
| `expires_at` | DATETIME NOT NULL | トークン有効期限（招待から30日） |
| `created_at` | TIMESTAMP DEFAULT CURRENT_TIMESTAMP | 招待発行日時 |

**インデックス**：
- `UNIQUE KEY uk_token (token)`
- `INDEX idx_status (status)`
- `INDEX idx_artist_post (artist_post_id)`

**status値の意味**：
- `invited` — 招待発行済み、画家未入力
- `submitted` — 画家入力完了、管理者レビュー待ち
- `partial_approved` — 一部承認・一部差戻し中
- `draft_saved` — 承認完了、artist投稿の下書き保存済み（**公開はまだ**）
- `rejected` — 全項目差戻し

### `wp_boa_artist_field_reviews` テーブル

項目ごとの承認状態を記録。差分承認のため。

| カラム | 型 | 説明 |
|---|---|---|
| `id` | BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY | レビューID |
| `submission_id` | BIGINT UNSIGNED NOT NULL | 提出ID（外部キー） |
| `field_key` | VARCHAR(100) NOT NULL | フィールド識別子 |
| `is_public_field` | TINYINT(1) NOT NULL DEFAULT 0 | このフィールドが公開対象か（本名等は0） |
| `status` | ENUM('pending','approved','rejected') DEFAULT 'pending' | 項目状態 |
| `original_value` | LONGTEXT DEFAULT NULL | 既存値（差分表示用） |
| `submitted_value` | LONGTEXT DEFAULT NULL | 画家提出値 |
| `final_value` | LONGTEXT DEFAULT NULL | 承認後の最終値（管理者編集可） |
| `reject_reason` | TEXT DEFAULT NULL | 差戻し理由 |
| `reviewed_at` | DATETIME DEFAULT NULL | レビュー日時 |

**インデックス**：
- `INDEX idx_submission (submission_id, field_key)`

**`is_public_field` の意義**：差分レビュー画面で「このフィールドは公開対象」「このフィールドは非公開（管理用）」を視覚的に区別するためのフラグ。本名・住所・振込先などは `0`、活動名・略歴・制作テーマなどは `1`。

### マイグレーション

プラグイン有効化フックで `dbDelta()` 使用。

## 4. トークン付URL設計

### URL形式

```
https://bankof-art.com/artist-onboarding/?token={64文字のランダムトークン}
```

### トークン生成

- `wp_generate_password(64, false, false)` で英数字64文字を生成
- 衝突チェックしてUNIQUE保証
- 有効期限：発行から30日（DB `expires_at`）

### トークン検証ロジック

1. URLパラメータ `token` を取得
2. DBから該当レコード取得
3. 以下を順にチェック：
   - レコード存在
   - `expires_at` を過ぎていない
   - `status` が `invited` または `rejected`（差戻し時の再入力許可）
4. 不正・期限切れの場合は「無効なリンク」画面を表示

### セキュリティ

- トークンはGETパラメータで渡すが、HTTPS必須
- 同一トークンで複数回フォーム開いても問題なし（送信は1回のみ）
- IPアドレスを `wp_options` に記録（万一の悪用調査用）
- レートリミット：同一IPから5分以内に10回以上アクセスでブロック

## 5. 画面設計

### URL構成

#### 画家側

- `GET /artist-onboarding/?token=xxx` — 入力フォーム
- `POST /wp-json/boa-artist/v1/submit` — 提出API（AJAX）
- `GET /artist-onboarding/complete/` — 送信完了画面
- `GET /artist-onboarding/expired/` — 期限切れ画面
- `GET /artist-onboarding/already-submitted/` — 提出済み画面

#### 管理者側

- WP管理画面 → **BOA画家管理** メニューを追加
  - 招待管理（一覧・新規招待）
  - 提出レビュー（差分承認UI）
  - 承認済み一覧（**下書き保存済み・公開済み**の状態が一目でわかる）

### 画家入力フォームのUI

#### 全体構成

- 既存BOAサイトのデザイントーンに統一
- フォームは縦1列、項目グループごとにセクション分け
- 進捗バー：入力済み項目数 / 必須項目数を上部に表示
- 自動保存：30秒ごとに入力内容を localStorage に退避（誤閉じ防止）
- 送信前確認モーダル：「送信後は画家本人からは編集できません」と明記

#### 入力グループの順序

1. はじめにのご案内
2. **公開項目セクション**（薄緑アクセント）
   - 基本情報（活動名・略歴）
   - 制作テーマ・なぜ描くか
   - SNSリンク
3. **非公開項目セクション**（薄グレーアクセント・「🔒 BOA内部のみ」表示）
   - 氏名（本名）
   - 契約情報すべて
4. 同意事項
5. 送信ボタン

#### バリデーション

- HTML5標準（required, type=email等）＋ JS追加バリデーション
- 送信前にすべてのエラーを画面上部にまとめて表示
- 文字数オーバーは入力時にリアルタイム表示

### 管理画面のUI

#### 招待管理画面

メニュー：BOA画家管理 → 招待

- 一覧テーブル
  - 招待日時 | 招待先名 | メール | ステータス | 残り日数 | アクション（再送・無効化）
- 新規招待ボタン → モーダルで「氏名・メール」入力 → トークン生成 → メール送信
- 検索：名前・メール・ステータスでフィルター

#### レビュー画面（差分承認UI）

メニュー：BOA画家管理 → 提出レビュー

**レイアウト**：左右2カラム
- 左カラム：項目一覧
  - 各項目に「🌐 公開項目」または「🔒 非公開項目」のラベル
  - 既存値 / 提出値 / 状態（差分表示）
- 右カラム：プレビュー（実際のARTIST個別ページ風）
  - **非公開項目はプレビューに表示しない**（実際の公開時と同じ表示）

**項目ごとのアクション**：
- ✓ 承認（提出値で確定）
- ✏️ 編集して承認（管理者が手直しした値で確定）
- ✕ 差戻し（理由を入力、画家にメール通知）

**一括操作**：
- 「すべて承認」ボタン
- 「すべて差戻し」ボタン（理由必須）

#### 承認完了時の挙動（重要）

**全項目が承認された時の処理**：

1. `final_value` を `artist` カスタム投稿のACFフィールドに `update_field()` で反映
2. 新規画家の場合：**`wp_insert_post()` で投稿作成（`post_status='draft'` 固定）**
3. 既存画家の場合：**`wp_update_post()` で更新（`post_status` は変更しない＝下書きなら下書きのまま）**
4. submission の `status` を `draft_saved` に更新
5. 管理者に通知メール送信：
   - 件名：【下書き保存完了】{画家氏名}様のARTIST投稿が下書きに保存されました
   - 本文：「プロフィール情報を下書き保存しました。プロフ写真・作品画像を追加のうえ、公開操作をお願いします」+ artist投稿の編集画面URL
6. **画家には「承認完了メール」は送らない**（まだ公開されてないため誤解を招く）

#### 公開操作（手動）

管理者がWordPress標準の投稿編集画面（`/wp-admin/post.php?post={id}&action=edit`）で：

1. 下書き状態のartist投稿を開く
2. プロフ写真・作品画像をメディアライブラリから追加
3. 全項目を改めて目視確認（**特に「氏名」が公開項目に紛れていないか**等）
4. WordPress標準の「公開」ボタンをクリック

公開検知用フック：

- `transition_post_status` フックでartist投稿が `draft → publish` になったタイミングを検知
- 該当する submission の `artist_post_id` から submission を特定し、画家に **公開完了通知メール** を送信

#### プレビュー機能

- ARTIST個別ページのテンプレート（`artist-single.html`相当）を一部簡略化したプレビュー用テンプレートを用意
- 画家が提出した値を流し込んで表示
- **公開対象項目のみ表示**（非公開項目は表示しない）
- 画像は管理者が後から追加するため、プレースホルダー表示

#### 承認済み一覧画面

メニュー：BOA画家管理 → 承認済み一覧

submissionテーブルで `status='draft_saved'` のもの一覧。

| 画家氏名 | 活動名 | 下書き保存日 | 画像追加状況 | artist投稿リンク | 公開状態 |
|---|---|---|---|---|---|
| 山田太郎 | YAMA | 2026-06-10 | ⚠️未設定 | [編集] | 下書き |
| 佐藤花子 | SATO | 2026-06-08 | ✓設定済 | [編集] | 公開済 |

**画像追加状況の警告**：プロフ写真・作品画像が未設定の投稿には「⚠️ 画像未設定」と警告表示し、公開前に画像追加を促す。

## 8. 実装ステップ（Claude Code向け作業順）

### Step 1：プラグイン雛形＋DB

- メインPHPファイル
- 有効化フックで2テーブル作成
- 無効化フックでオプション削除（DBデータは保持）

### Step 2：トークン管理

- トークン生成関数（衝突チェック）
- トークン検証関数
- URLルーティング（`/artist-onboarding/`のリライト）

### Step 3：画家入力フォーム実装

- 入力フォームテンプレート（HTML/CSS、BOAデザイン準拠）
- 公開項目セクションと非公開項目セクションの視覚分離
- JS：HTML5バリデーション、文字数カウント、自動保存
- REST API `/submit` エンドポイント
  - バリデーション
  - DB INSERT（`submitted_data` にJSON保存、`status='submitted'`）
  - `wp_boa_artist_field_reviews` に項目ごとレコード作成（`is_public_field` フラグ設定）
  - 完了メール送信（画家・管理者）
  - 完了画面リダイレクト

### Step 4：管理画面（招待管理）

- 招待一覧テーブル
- 新規招待モーダル → トークン生成 → メール送信
- 再送・無効化ボタン

### Step 5：差分承認UI

- レビュー画面の左右2カラムレイアウト
- 公開/非公開フラグの視覚表示（🌐 / 🔒）
- 既存値 vs 提出値の差分表示（色分け：既存=グレー、提出=緑、変更=黄色）
- 項目ごとの承認/編集/差戻しボタン（AJAX）
- 一括操作

### Step 6：プレビュー機能

- ARTIST個別ページの簡略テンプレート
- 提出データを流し込んでiframe表示
- **公開項目のみ表示**（非公開項目は表示しない）
- 「画像はプレースホルダー」表示

### Step 7：下書き保存処理（重要：公開しない）

- 全項目承認時に `final_value` を `update_field()` でartistカスタム投稿に反映
- **`post_status='draft'` で固定**（既存なら現在のステータスを維持＝下書きなら下書きのまま、公開済みの場合も自動公開はしない）
- 管理者に「下書き保存完了通知メール」送信
- **画家には通知しない**

### Step 8：公開検知＆通知

- `transition_post_status` フックを実装
- artist投稿が `draft → publish` になったとき、該当 submission を特定して画家に公開通知メール送信
- 公開検知のログ出力

### Step 9：承認済み一覧画面

- `status='draft_saved'` の一覧表示
- 画像未設定の警告表示
- artist投稿編集画面への直リンク

### Step 10：テスト

- 招待〜提出〜承認の一連フロー
- 承認後に下書き保存されることを確認（**公開されないこと**）
- 管理者が手動で「公開」ボタンを押すと画家に通知メールが届くこと
- 期限切れトークンの挙動
- 差戻し→再入力の挙動
- 不正なトークンでアクセス時の挙動
- 公開項目／非公開項目の視覚区別が正しく機能すること
- プレビュー画面に非公開項目が表示されないこと

## 9. セキュリティ・運用考慮

### トークンセキュリティ

- 64文字英数字ランダム（推測困難）
- HTTPS必須
- 30日有効期限
- 1度承認されたらトークンは無効化（再入力には新規トークン発行）

### CSRF・XSS対策

- REST APIに `wp_create_nonce()` + `X-WP-Nonce` 検証
- 入力値は `esc_html()` `esc_attr()` `esc_url()` でエスケープ
- HTMLタグはストリップ（textarea以外）

### スパム対策

- ハニーポット（hidden text field）
- reCAPTCHA v3（管理画面でサイトキー設定可、無効化も可）
- 同一IPから10分以内に5回以上の送信失敗でブロック

### 個人情報（最重要）

- 契約情報は管理画面で `manage_options` 権限のみ閲覧可
- DB値は暗号化しない（運用上必要）が、サイト上には絶対表示しない
- プレビュー画面に非公開項目は出さない
- artist投稿のACFフィールドで、非公開フィールドは `display=false` 設定
- プラグイン無効化時に契約情報を削除するかどうか設定可能

### 公開事故防止策（多重防御）

1. **承認 = 下書き保存のみ**（自動公開しない）
2. **公開項目／非公開項目を視覚分離**（フォーム・管理画面・プレビュー全てで）
3. **承認完了メールは公開時にのみ送信**（誤承認で画家を混乱させない）
4. **承認済み一覧画面で画像未設定を警告**（不完全な状態での公開を防ぐ）
5. **公開時のフック検知**で画家通知メールを自動送信（手動公開を確実に画家に伝える）

### ログ

- 招待送信
- 入力提出
- 承認/差戻し
- ACF反映成功/失敗
- 公開操作検知
- メール送信成功/失敗
- WordPress標準のerror_logに出力（WP_DEBUG_LOG有効時）

## 12. ACFフィールドマッピング

### 公開フィールド

| フォームフィールド | ACFフィールド名 | 公開 |
|---|---|---|
| 活動名 | `artist_name` | 🌐 |
| 略歴 | `artist_bio` | 🌐 |
| 制作テーマ（13字以内） | `artist_theme` | 🌐 |
| 制作テーマ詳細 | `artist_theme_detail` | 🌐 |
| なぜ描くか | `artist_reason` | 🌐 |
| Instagram | `artist_sns_instagram` | 🌐 |
| X | `artist_sns_x` | 🌐 |
| Facebook | `artist_sns_facebook` | 🌐 |
| YouTube | `artist_sns_youtube` | 🌐 |
| その他URL | `artist_sns_other` | 🌐 |

### 非公開フィールド（管理者専用）

| フォームフィールド | ACFフィールド名 | 公開 |
|---|---|---|
| 氏名（本名） | `artist_legal_name` | 🔒 |
| 本名フリガナ | `artist_legal_name_kana` | 🔒 |
| 生年月日 | `artist_birthdate` | 🔒 |
| 郵便番号 | `artist_postal_code` | 🔒 |

| 連絡先メール | `artist_contact_email` | 🔒 |
| 振込先金融機関 | `artist_bank_name` | 🔒 |
| 支店名 | `artist_bank_branch` | 🔒 |
| 口座種別 | `artist_bank_account_type` | 🔒 |
| 口座番号 | `artist_bank_account_number` | 🔒 |
| 口座名義 | `artist_bank_account_holder` | 🔒 |

**重要**：非公開フィールドはACF設定で必ず以下を行うこと：
- `Conditional Logic` で表示制御
- `Permissions` で管理者のみ閲覧可
- カスタムフィールド出力時には絶対にテンプレートに含めない（`get_field()` を使う場所を限定）

ACFフィールド名は既存スキーマと一致させる必要があるため、Claude Code実装前にACFエクスポートを確認すること。

## 13. Claude Code 実装時の補足指示

- 既存の `artist` カスタム投稿（ACF）のフィールド名を `wp option get` または `wp acf export` で確認してから実装
- フォームHTMLは既存BOAサイトのデザイントークン（フォント・色・余白・ボタンスタイル）を使う
- 差分表示UIはWordPress標準のRevision比較風（左右2カラム、変更箇所をハイライト）
- プレビューはiframeで `/artist-onboarding/preview/?submission_id={id}` を読み込む形が簡単
- **重要**：承認処理は必ず `post_status='draft'` 固定。誤って `publish` にしないこと
- 単体テスト不要、結合テストで以下のフローを通せばOK：
  1. 招待メール送信
  2. 画家がフォーム入力・送信（公開/非公開項目の視覚分離を確認）
  3. 管理者が差分レビュー（一部承認・一部差戻し）
  4. 画家が再入力・再送信
  5. 全項目承認 → **下書き保存される（公開されない）こと**を確認
  6. 管理者が手動で公開 → 画家に通知メールが届くことを確認
- 完成後、README に「招待方法」「レビュー手順」「公開手順」「ACFフィールド対応表」を記載
